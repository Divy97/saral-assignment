⏺ The single-flag approach is right. Two real issues, one non-issue, and one
  rule to keep it clean.

  The flag should branch exactly once

  STAGE reads in one place — a composition root / factory that builds the
  concrete Queue, Storage, Scheduler and injects them. If if (stage === 
  'production') shows up scattered across services, you've lost the whole
  benefit. Everything downstream takes the interface, never reads the env. Get
  this wrong and the adapter seam is theater.

  Issue 1 — queue semantics (you're right to worry)

  This is the only genuinely hard divergence:

  ┌──────────────────────┬────────────────────────┬─────────────────────────┐
  │                      │       In-memory        │           SQS           │
  ├──────────────────────┼────────────────────────┼─────────────────────────┤
  │ Delivery             │ exactly-once (same     │ at-least-once — dupes   │
  │                      │ process)               │ happen                  │
  ├──────────────────────┼────────────────────────┼─────────────────────────┤
  │ Consumer             │ in-process function    │ poll loop / Lambda      │
  │                      │ call                   │ trigger                 │
  ├──────────────────────┼────────────────────────┼─────────────────────────┤
  │ Retry / visibility   │ none unless you build  │ built-in                │
  │ timeout              │ it                     │                         │
  ├──────────────────────┼────────────────────────┼─────────────────────────┤
  │ Failure              │ lost on crash          │ redriven, DLQ           │
  └──────────────────────┴────────────────────────┴─────────────────────────┘

  Design the interface to the lowest common denominator: enqueue(msg) is
  fire-and-forget async, and a consumer polls (receive → handle → ack/delete).
  The local version runs an in-process poll loop over an array so it behaves
  like SQS, not like a direct call. If you make local a synchronous await 
  handler(), prod will surprise you.

  The dupe problem solves itself: make the handler idempotent — INSERT ... ON 
  CONFLICT (media_id) DO UPDATE. You need that anyway for the "avoid duplicate
  media" requirement. So SQS at-least-once costs you nothing extra. Two
  requirements, one mechanism.

  Issue 2 — the scheduler is NOT an SDK swap

  node-cron runs inside your long-lived Express process. EventBridge→Lambda is a
  different execution topology invoked from outside — not something you toggle
  with an SDK client in the same running app. So be honest about what
  "production" means:

  - Cleanest: production still = one Express process, just with SQS + S3
  adapters. node-cron stays in both. You've structured for Lambda (the sync job
  is a plain function the cron calls) without pretending a flag deploys it.
  - Don't wire up an actual Lambda/EventBridge deploy unless the reviewers ask.
  That's infra work the assignment doesn't need, and it can't run in Docker
  Compose anyway.

  Say this plainly in instructions.md under tradeoffs: "Scheduling is node-cron
  in both stages; the sync job is isolated as a handler so it drops into Lambda
  unchanged." That reads as judgment, not a gap.

  Non-issue — storage

  Correct, this is trivial. Interface is put(key, buffer, contentType) → url.
  Local writes to ./media and returns http://localhost/media/...; S3 uploads and
  returns the object URL. Clean swap, no semantic gap.

  Net

  Single STAGE flag: yes. Guardrails:
  1. Flag branches in one factory; everything else is injected.
  2. Queue interface = async enqueue + polling consumer (local mimics SQS, not
  the reverse).
  3. Idempotent upsert on media_id — covers dupes and SQS redelivery.
  4. node-cron in both stages; don't fake a Lambda deploy. Document it as
  deliberate.

  Want the interface signatures + the factory sketch next?

⏺ Your instinct is correct — split it. Fetching metadata and transferring
  binaries are different failure domains and shouldn't share a fate. Here's the
  design I'd commit to, with two refinements and one gotcha you haven't hit yet.

  Two job types

  - SYNC_RECENT (fan-out): paginate Meta → upsert metadata rows with
  asset_status = 'pending' → for each new/incomplete row, enqueue a FETCH_ASSET.
  Fast, all-or-nothing on metadata only.
  - FETCH_ASSET (per media): download from Meta's media_url → upload to storage
  → UPDATE row with the storage URL + asset_status = 'done'. Fails in isolation,
  retries in isolation.

  Metadata lands immediately, so the read API returns rows right away with the
  asset URL filling in async. Good.

  This quietly resolves the created_at question you parked

  The metadata INSERT sets created_at once. The asset job is an UPDATE that
  touches updated_at, never created_at. So the read API ordering by created_at 
  DESC stays stable no matter how many times the row gets updated. The two-phase
  split is the answer to that earlier worry.

  Idempotency (mandatory — SQS is at-least-once)

  - Metadata: upsert on media_id.
  - Asset job: deterministic storage key (media/{media_id}.{ext}) so a re-run
  overwrites the same object — no orphans. And skip early if asset_status = 
  'done' already.
  - Re-sync every 3h overlaps heavily on recent_media; only enqueue FETCH_ASSET
  for rows that are new or not done. Cheap, avoids re-downloading everything
  each cycle.

  DLQ / retries

  - Production: free — SQS maxReceiveCount → DLQ. Zero code.
  - Local: don't build a DLQ system. A receiveCount on the in-memory message +
  asset_status = 'failed' after N tries is enough. // ponytail: naive retry 
  counter, SQS handles this natively in prod.

  The gotcha you'll hit: Instagram media_url expires

  Meta's media_url is a short-lived signed CDN link. If a FETCH_ASSET job sits
  in the queue or DLQ too long, the URL is dead and no amount of retrying
  downloads it — the retry fails identically. Two implications:

  1. Keep pending→download latency short. Per-media jobs enqueued right after
  the metadata upsert, not batched for later.
  2. On a dead URL, retrying the download is pointless; the correct recovery is
  re-fetching the media's media_url from Meta first. For a take-home I'd not
  build that — just let it hit failed/DLQ and note it in tradeoffs. Flagging it
  is the judgment they're grading; building the refresh path is scope you don't
  need.

  Net pipeline

  cron/EventBridge (3h)
     └─ enqueue SYNC_RECENT
          └─ worker: paginate Meta → upsert metadata (pending)
               └─ per new/incomplete media: enqueue FETCH_ASSET
                    └─ worker: download → upload → update row (done)
                         └─ on fail: retry → DLQ → asset_status=failed

  One open question this surfaces: does a failed asset still show in the read 
  API? I'd say yes — return the row with metadata + permalink and a null storage
  URL, rather than hiding it. The record exists; the binary just didn't
  transfer. Agree, or should GET /hashtags only return done rows?

  Watermark — you're right, drop it

  Yes: a watermark = "timestamp of the newest post from the last sync," used to

  - Dedup and no-re-download are already guaranteed by upsert + status, exactly
  as you said. Correctness needs zero extra state.
  - The early-stop is a pure API-call optimization, and it's moot in both volume
  regimes:

  paginate back to already-seen posts → early-stop never triggers.

  anyway.

  already in the media table — storing a separate watermark column is
  denormalized state that can drift out of sync with reality. Classic thing to
  not persist.

  A bare timestamp cursor breaks keyset pagination whenever rows share the
  ordering value — and with our design that's the common case, not an edge one:

  - created_at defaults to now(), and in Postgres now() returns the transaction 
  start time — identical for every row inserted in the same statement.
  - We insert per page, in batches. So an entire page of media gets the same
  created_at.
  - With a timestamp-only cursor: WHERE created_at < :ts skips the rest of that
  same-timestamp batch on the next page; <= re-returns rows you already saw.
  Guaranteed dup-or-skip at every batch boundary.

  The tiebreaker (media_id) is what makes it correct, so the cursor has to carry
  both values. Two ways to keep your next_cursor field and stay correct:

  - Recommended — next_cursor stays an opaque token encoding (created_at, 
  media_id), base64url. Same field name, client just echoes it back. This is
  what Stripe/Slack cursors are; "timestamp-ish" but safe.
  - If you want a literal timestamp value for readability: switch created_at to
  clock_timestamp() (per-row microsecond time, not transaction time) so rows are
  effectively unique. Still carries a small collision risk under concurrent
  inserts, so I'd only take this if the opaque token is a dealbreaker.

  My call: keep next_cursor as the opaque token. You get the rename,
  correctness, and a client that can't construct an inconsistent cursor. Want it
  that way, or do you specifically want the raw-timestamp version with
  clock_timestamp()?
