⏺ Both are fair questions, but the Dockerfile is actually correct on both
  counts. Let me explain rather than change it.

  1. corepack enable before COPY — this is fine, not broken

  corepack enable just installs the pnpm/yarn shims onto the PATH. It reads
  nothing from package.json and needs no project files — it's pure environment
  setup on the base image.

  The packageManager field (pnpm@10.24.0) is read later, when pnpm install runs
  — and by then package.json has been copied. So the resolution happens at the
  right moment. Enabling corepack early is the standard order precisely because
  it's image setup, independent of your code. If I flipped it (COPY then
  corepack enable), nothing would improve and I'd just lose a tiny bit of layer
  caching.

  2. No tsc build — deliberate, because we run via tsx

  The start script is ... && tsx src/entrypoints/local.ts. tsx executes
  TypeScript directly at runtime (esbuild under the hood) — no compile step, no
  dist/. So a tsc build in the image would produce output we never use.

  tsc still earns its keep as pnpm typecheck (in CI/pre-commit) — that's type
  checking, separate from running. We just don't need a build artifact to boot.

  When would you build in the Dockerfile? A real production deploy — you'd tsc
  to dist/, install only prod deps, and drop tsx for faster cold starts and a
  smaller image. We explicitly aren't deploying (local stage, reviewers run it),
  so tsx-run is the lazy-correct call. // ponytail: tsx runs TS directly; no 
  build artifact needed for the local container.

  If you'd rather ship a compiled image anyway — it's a small change:
  multi-stage build, RUN pnpm build, copy dist/, CMD ["node", 
  "dist/entrypoints/local.js"], and move tsx out of the run path. Want that
  version, or keep tsx for simplicity?

⏺ Two things — first a correction, then the brief you asked for, then I'll fix
  it properly.

  Correction: there's no temp+rename today

  Current put is a single await writeFile(target, body) — no temp file, no
  rename. So there's no two-step gap right now. But your instinct is good,
  because a single writeFile of a big buffer isn't atomic either (a crash
  mid-write leaves a truncated file), and it has the memory problem you're
  sensing.

  writeFile(buffer) internals — the brief

  - It's async via libuv's threadpool — it does not block the event loop.
  - It writes the buffer you hand it in chunks, but the entire asset is already 
  resident in memory as that Buffer. writeFile can't make it smaller — you
  allocated the whole thing before calling it (and again upstream, when the
  download did arrayBuffer()).
  - So the memory cost isn't writeFile itself — it's holding the whole file as a
  Buffer. A 100 MB video = a 100 MB buffer in RAM for its whole lifetime.
  Single worker → one at a time, so peak ≈ largest single asset. Usually fine,
  but unbounded by file size.

  Better alternative: stream end-to-end + temp/rename (gives you both)

  Stream the download straight to disk — never hold the whole file in memory —
  and publish via temp+rename so a reader never sees a partial file:

  - Memory-bounded: pipeline(stream, createWriteStream(tmp)) copies
  chunk-by-chunk.
  - Atomic: rename(tmp, target) is atomic on the same filesystem — the file
  appears complete or not at all.

  The catch: the port must take a Readable stream, not a Buffer. Cheap to change
  now (no consumers built yet). Making that change:

⏺ Exactly the right thing to poke at. The issue is specifically about the temp 
  file name, which is fixed: 123.jpg.tmp, derived only from the key.

  Single worker (now): only one put for a given key runs at a time, so nothing
  else ever touches 123.jpg.tmp while it's being written. Safe.

  Multiple workers (later): the same media's FETCH_ASSET can be handled by two
  workers at the same time. This isn't hypothetical — it's normal SQS behavior:
  - At-least-once delivery: if worker A is slow (big video) and its visibility
  timeout expires, SQS redelivers the same message to worker B while A is still
  going.
  - Or a duplicate enqueue for the same media_id.

  Now both call put("123.jpg", …) → both open and write the same 123.jpg.tmp
  concurrently:

  worker A: ...writes bytes → 123.jpg.tmp
  worker B: ...writes bytes → 123.jpg.tmp   (interleaved!)
  → temp file is a garbled mix of both streams
  → rename publishes that garbage as 123.jpg

  The whole point of temp+rename is that the temp is a complete copy before the
  atomic rename. A shared temp name breaks that — two writers corrupt the temp,
  so the "atomic" publish now atomically publishes corruption. The rename is
  still atomic; the file it renames is the problem.

  The fix when you scale: make the temp unique per write, not per key —

  const tmp = `${target}.${randomUUID()}.tmp`;

  Then each worker streams into its own temp; whoever renames last wins, and
  since it's the same media, the content is identical — last-writer-wins is
  harmless. Each published file is always a complete copy.

  Recommendation: leave it as-is now. Single worker makes it correct, and the
  comment marks the exact one-line upgrade for when we add workers. That's the
  ponytail move — don't build for a worker count we don't have yet. If you'd
  rather pay the one line now to remove the future footgun entirely, it's
  genuinely trivial (randomUUID from node:crypto) — your call.

❯ Id as string would work ? we decided that it would be number, right ?         

⏺ Actually the opposite — we decided string, and it's the right call. I think
  the mix-up is bigint (DB) vs number (JS); we deliberately avoided both.

  Three reasons id: z.string() is correct:

  1. Meta literally sends it as a string. In your samples the value is quoted:
  "id": "18130665154527971". JSON string → z.number() would reject the real
  response outright.
  2. We locked the DB column as TEXT, not bigint — "Meta ids are opaque strings;
  don't assume they stay numeric or in range." The migration has media_id TEXT 
  PRIMARY KEY. String on the wire maps straight to TEXT.
  3. A JS number would silently corrupt it. 18130665154527971 ≈ 1.8×10¹⁶, which
  is past Number.MAX_SAFE_INTEGER (2⁵³ ≈ 9×10¹⁵). Parse it as a number and you
  get 18130665154527972 — off by one. That's a real dedup bug: the id you store
  wouldn't match the id Meta sends next time.

  So z.string() stays. No change needed. (What we called "bigint, no" back in
  the schema talk was exactly this — keep the id as an opaque string end to
  end.)

  fill in the URL, flip asset_status to pending, and enqueue a FETCH_ASSET.

  overwrite an existing one.

⏺ Local single-worker blocking is what surfaced it, but no — it's not the only
  reason, and syncs still shouldn't be queue jobs in production even with
  concurrent Lambda consumers. Three independent reasons:

  1. SQS visibility timeout vs a long-running sync. An in-flight SQS message has
  a visibility timeout; if the handler runs longer than it, SQS assumes the
  consumer died and redelivers the message — so a second sync starts while the
  first is still paginating. A sync can run minutes (up to 500 items × pages ×
  Meta latency), so you'd be fighting this with long timeouts + heartbeating.
  Assets are short, bounded, single-shot — a natural fit. Long fan-out
  orchestration in an SQS-triggered handler is a known anti-pattern for exactly
  this.

  2. A sync is a scheduled trigger, not a work item. It fires on a clock (every
  3h) from one producer (the scheduler). The queue's value — buffering bursts,
  per-item retry, DLQ, decoupling many producers from many consumers — buys
  nothing for a once-per-3h singleton. EventBridge → sync Lambda directly is the
  right driver; EventBridge → SQS → Lambda is a pointless extra hop.
  (EventBridge itself has retry + DLQ for the invocation if you need
  durability.)

  3. Opposite concurrency needs. You want at most one sync at a time — two
  concurrent SYNC_RECENT double-fetch and burn Meta rate limit. You want high
  concurrency for assets — parallel downloads. "Concurrent consumers" is the
  right answer for assets and the wrong answer for syncs. One queue with one
  concurrency setting can't serve both; you'd need a separate sync queue pinned
  to concurrency=1 (FIFO/dedup) — which is just reintroducing complexity to
  emulate "don't queue it."

  So the real asymmetry: assets are the textbook queue workload (many, bursty,
  idempotent, per-item retry) — and the sync is the orchestration that produces 
  that workload. Producer triggered by a schedule; consumer fed by a queue. That
  split is correct at every scale, local or Lambda — the single-worker pain
  just made it visible early.

  Could you force both into queues with per-queue concurrency configs?
  Technically yes — but it's more moving parts to work around a mismatch, with
  no upside over triggering the sync directly. Not worth it.

⏺ Right on the numbers — SQS default visibility is 30s (tunable up to 12h), and
  a minutes-long paginating run would blow past a sane value and get redelivered
  mid-flight. Good instinct.

  And your scaling fix is the correct pattern: cursor-chained page jobs. Instead
  of one long orchestration, a SYNC_PAGE job carries the after cursor, does one
  page (fetch → upsert → enqueue that page's FETCH_ASSETs), then enqueues the
  next SYNC_PAGE with the new cursor. Each unit is short (well under any
  visibility timeout), retries per-page, and it scales to arbitrary fan-out.
  That's the standard "continuation via re-enqueue" shape, and the after cursor
  is exactly the continuation token for it.

  One non-obvious cost to bank for if you ever build it — it's more than just
  "split the loop": at-least-once means a page job can be redelivered, and on 
  redelivery it re-enqueues the next page → the chain forks into two parallel
  paginations double-fetching. So you'd need SQS FIFO + content dedup (or a
  "current cursor" guard in the DB) to keep the chain single. That's the real
  complexity the continuation model adds, on top of the same at-most-one-chain
  concern we already have.

  And your conclusion is the right one: for ~500 items — a handful of pages —
  one direct run finishes in seconds, nowhere near any timeout. Chaining trades
  real simplicity for scale we don't have. Correctly deferred.

  Want me to jot this as the documented scaling path in tradeoffs (like we did
  for single-worker and the temp-file collision)? Otherwise this is just a good
  conversation and nothing changes in the code.
